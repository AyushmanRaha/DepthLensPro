      DepthLens Pro field manual

      Learn the renderer, workflows, metrics, and 3D depth language in one place.

      Use this self-contained guide when you are choosing a model, tuning real-time webcam depth, comparing engines, validating against ground truth, or exporting a point cloud. Nothing in this tab calls the backend; it is a local reference for confident, repeatable depth work.

      Scrollable header tabs Local-first inference flow Metric and glossary reference 3D reconstruction checklist

      ▣ Guide

      The header tabs switch workspaces; if they overflow, scroll the tab bar with a wheel, trackpad, touch, or drag gesture. The engine status orb reports backend health, the theme toggle switches dark/light presentation, and cards group controls by task.

      1. Start in Workspace for normal image depth inference.
      2. Watch disabled buttons and progress bars: they prevent duplicate work while uploads or inference are in flight.
      3. Use toasts for recovery hints; the app is local-first, so images stay in the local renderer/backend flow unless you export them.
      4. Use GUIDE anytime nomenclature or metrics are unclear.

      Buttons, sliders, and guide controls are intentionally no-drag zones inside the Electron draggable header/body so clicks remain precise.

      Workspace turns an image queue into depth maps. Pick Auto device selection for the best detected accelerator, CPU for compatibility, or a GPU path such as CUDA, MPS, XPU, or NPU / Neural Engine when available.

      1. Choose a compute device and model architecture.
      2. Upload one or more images to the image queue.
      3. Select a colormap to convert single-channel depth into readable color.
      4. Optionally attach ground truth depth to compute GT-aware metrics.
      5. Read the session dashboard for latency, cache hits, image dimensions, and metric summaries.

      MiDaS Small is best for quick CPU tests and webcam previews. DPT Hybrid balances quality and speed. DPT Large favors detail and usually needs a strong GPU for practical latency.

      Webcam mode asks for camera permission, captures frames, and runs real-time monocular depth estimation. Target FPS caps protect the backend from overload, frame max dimension reduces compute, and smoothing stabilizes flicker at the cost of a little responsiveness.

      Start with MiDaS Small, Auto/CPU, a modest max dimension, and low target FPS. Increase quality only after latency is stable.

      Compare runs one image through the available model set so you can inspect depth boundaries, object separation, and speed. Use it when deciding whether a scene needs DPT Large detail or whether MiDaS Small/DPT Hybrid is sufficient.

      Charts help separate quality-vs-speed decisions: visual sharpness can improve while latency and memory cost rise. When ground truth is present, prefer lower Abs Rel/RMSE and higher δ < 1.25 or SSIM.

      Performance compares PyTorch execution with optional ONNX Runtime. ONNX can be unavailable if weights, dependencies, or a matching provider are missing. Provider means the runtime backend, such as CPU or CUDA; fallback means the app safely returns to another provider/path.

      LatencyTime for one inference; lower feels snappier.ThroughputImages or frames per second; higher supports batch/live workloads.SpeedupRelative improvement over a baseline.RSS memoryResident process memory while benchmarking.

      Experiments package repeated validation into named runs. Use GT runs when ground truth depth is aligned with each image; use image-only runs for qualitative batches. JSON exports preserve detailed metadata, while CSV exports are convenient for spreadsheets and reports.

      For reproducibility, record model, device, input size, colormap, timestamps, cache state, and any median-scale alignment used for relative-vs-GT comparisons.

      Monocular depth is generally relative depth, not calibrated metric depth. Reconstruction converts one source image and its depth map into an approximate point cloud, useful for visualization and creative geometry—not survey-grade measurement.

      • PLY is a point-cloud friendly export; OBJ is broadly accepted by 3D tools.
      • Max points and preview points trade detail for responsiveness.
      • Grid sampling is evenly spaced; random sampling can preserve stochastic coverage.
      • Focal scale and depth scale shape perspective and z spacing.
      • Near/Far percentile clipping removes extreme depth outliers.
      • Coordinate system and RGB color inclusion determine orientation and whether points carry image color.

      Point cloud, focal scale, depth scale3D points and controls that shape projection and depth spacing.
      • Backend offline/degraded: check the engine orb, refresh status, and confirm dependencies are installed.
      • Unsupported device: switch to Auto or CPU; CUDA/MPS/XPU/NPU availability depends on hardware and runtime builds.
      • Large files or slow inference: reduce image size, use MiDaS Small, or choose a faster device.
      • GT mismatch: ensure ground truth dimensions and pairing match the source image.
      • Webcam permission failure: allow camera access and close other apps using the camera.
      • ONNX unavailable: use PyTorch or install/provide the correct ONNX weights/provider.

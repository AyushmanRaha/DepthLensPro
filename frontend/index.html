      DepthLens Pro field manual

      Learn the renderer, workflows, metrics, and 3D depth language in one place.

      Use this self-contained guide when you are choosing a model, tuning real-time webcam depth, comparing engines, validating against ground truth, or exporting a point cloud. Nothing in this tab calls the backend; it is a local reference for confident, repeatable depth work.

      Guide

      DepthLens Pro performs monocular depth estimation: it looks at one normal RGB image and predicts a relative near-to-far depth value for each pixel. The colored output is a depth map, not a second photograph.

      Use it for visual inspection, creative depth masks, comparing MiDaS/DPT models, local benchmarks, ground-truth validation, webcam previews, and approximate point-cloud exports.

      Important limitation: the prediction is relative depth. It is not a tape measure, LiDAR scan, stereo camera result, survey-grade mesh, or guaranteed metric reconstruction.

      Good useSee which objects are nearer, create depth-aware visuals, compare models, benchmark runtimes, preview approximate 3D.Wrong useMeasuring exact meters, safety-critical navigation, construction surveys, medical decisions, or replacing calibrated sensors.
      1. Check the engine status orb. Green/ready means the local backend is responding; offline or degraded means open the orb for diagnostics before starting.
      2. Choose Auto device so the backend can pick the safest available CPU/GPU path.
      3. Choose MiDaS Small for the first run because it is the fastest supported model and works well on CPU.
      4. Pick a colormap such as Inferno, Magma, or Viridis.
      5. Upload one PNG, JPG, WEBP, BMP, or other browser-supported image under the 20 MB backend limit.
      6. Click Generate Depth Maps and wait for the progress card to finish.
      7. Read the result card: dimensions, latency, engine used, cache state, and metrics describe what happened.
      8. Download the color depth map, grayscale depth, or metadata if you need files for another tool.

      Recommended first settings: Auto device · MiDaS Small · Inferno or Viridis · one medium-size image · metrics on the default mode · ground truth off.

      The Workspace is the normal image-to-depth pipeline. The image queue accepts multiple images for batch mode; GT mode intentionally narrows the run to one image plus one GT file so benchmark metrics are paired correctly.

      • Image Queue: drag, browse, clear, or cancel. Buttons disable while work is running to prevent duplicate requests.
      • Progress: bars and ETA text track upload/inference progress; cancelled requests stop UI processing when supported by the browser.
      • Cache: repeated image-only requests can return cache hits when model, colormap, device, metrics, outputs, max dimension, and image content match. GT uploads bypass cache.
      • Gallery: result cards show the original/depth preview, metrics, cache badges, and download actions.
      MiDaS SmallFastest; best for CPU, webcam, testing, and large batches where responsiveness matters.DPT HybridBalanced model; use when MiDaS Small is too coarse but DPT Large is too slow.DPT LargeMost detailed supported model; slower and usually practical only with a strong accelerator.Auto / CPU / acceleratorsAuto is safest. CPU is most compatible. CUDA, MPS/CoreML-class paths, XPU, or NPU appear only when the runtime detects support.

      If a device fails or disappears, switch back to Auto or CPU. Accelerator labels reflect local hardware, installed Python packages, and runtime provider availability.

      A colormap converts a single-channel depth plane into visible colors. The colors are a legend for depth values, not real-world object colors. Depth planes are normalized per image, so the brightest or hottest color means high relative value within that image.

      Inferno / MagmaHigh contrast and readable on dark backgrounds; good default inspection palettes.ViridisPerceptually uniform and more colorblind-friendly; good for reports.Plasma / Turbo / JetVery punchy contrast; useful for spotting structure but can exaggerate boundaries.Hot / BoneGrayscale-like or heat-style views; useful when color should be less distracting.

      Do not compare two separate images by color alone. Because normalization is per image, the same color may represent different absolute or relative ranges across scenes.

      Ground truth depth is a reference depth label from a dataset, sensor, simulator, or calibrated pipeline. Use GT mode when you have a source image and its matching depth label and want benchmark-style error metrics.

      • Supported GT files are PNG, TIFF/TIF, and numeric NPY, up to 20 MB.
      • Single-channel depth is preferred. Multi-channel image GT is converted to grayscale with a warning.
      • The backend resizes GT to prediction resolution with nearest-neighbor alignment to avoid inventing interpolated depths.
      • Median-scale alignment scales the prediction so its median valid depth matches the GT median, reducing monocular scale ambiguity before computing errors.
      • Optional GT scale and invalid-value settings help with units such as millimeters and sentinel pixels.

      Most bad GT scores come from mismatched image/GT pairs, wrong units, invalid pixels, unsupported files, or dimensions that do not correspond to the source image.

      No single metric tells the whole story. Use visual inspection for artifacts and boundaries, latency for usability, and GT metrics only when the reference file truly matches the image.

      Webcam mode requests camera permission, captures JPEG frames locally, and sends them to the local backend at a controlled target FPS. Frame max dimension controls compute cost, and smoothing blends recent depth frames to reduce flicker.

      • Backend latency is model processing time. End-to-end latency also includes capture, encoding, request, decode, and display.
      • Skipped frames are normal when a previous frame is still processing; they protect the backend from queue buildup.
      • MiDaS Small is usually best for webcam because consistent low latency often looks better than slow high detail.
      • Local-first privacy still applies: frames are processed by the local app/backend unless you export or share them.
      Stable CPUMiDaS Small · Auto/CPU · 1–2 FPS · 256–384 px · smoothing 0.25–0.5.Quality GPUDPT Hybrid or Large · accelerator · 2–5 FPS · 384–512 px · modest smoothing.Low-lagMiDaS Small · fastest device · low max dimension · low smoothing · accept less detail.

      Compare runs one image through all supported models using the selected device and colormap. It is the fastest way to decide whether a scene benefits from a slower model.

      1. Upload one representative image.
      2. Select a device and colormap.
      3. Run Compare and inspect results side by side.
      4. Look at object boundaries, thin structures, foreground/background separation, smoothness, and obvious artifacts.
      5. Use charts for speed/metric tradeoffs, but choose the model that serves the actual workflow.

      A faster model can be better than a higher-detail model for webcam, iteration, batch triage, low-power laptops, or any task where stale results are worse than slightly softer depth.

      The Performance tab has Benchmark and Observability views. Benchmark compares PyTorch with optional ONNX Runtime using a deterministic synthetic 384×384 frame, so no image upload is needed.

      PyTorchDefault inference path using Torch Hub MiDaS/DPT models.ONNX RuntimeOptional exported model path; requires local ONNX weights and compatible providers.ProviderThe execution backend such as CPU, CUDA, CoreML, OpenVINO, or another installed ORT provider.FallbackA safe return to PyTorch or CPU/provider path when ONNX/device support is missing.

      Observability shows local Prometheus-style metrics, runtime snapshots, inference latency history, bounded traces, crash analytics, cache events, and benchmark history. Use it to diagnose degraded readiness, slow requests, cache misses, unavailable ONNX files, or repeated errors.

      Experiments package repeated image-only or GT validation runs into named records. JSON exports keep detailed metadata; CSV exports are convenient for spreadsheets and reports.

      • For image-only experiments, keep model, device, colormap, max dimension, and cache policy consistent.
      • For GT experiments, verify every image has the correct GT file, invalid-value policy, unit scale, and median-scale alignment metadata.
      • Record timestamps, input sizes, model IDs, device/provider, cache state, latency, metrics mode, and exported outputs.
      • Run a small pilot before a large batch to catch unit or pairing mistakes.

      3D Reconstruction uses the source RGB image and predicted depth map to project colored points through a simple pinhole-camera approximation. It is useful for preview geometry, not measured shape.

      PLY / OBJPLY is point-cloud friendly; OBJ is widely accepted by 3D tools.Max dimensionReduces input size before depth/reconstruction to control memory and speed.Max / preview pointsLimit exported and viewer points; fewer points are faster, more points show more detail.Grid / randomGrid is deterministic and even; random samples deterministic coverage with stochastic distribution.Focal / depth scaleFocal changes perspective spread; depth scale stretches or compresses z spacing.Near / far percentilesClip outlier depth values before projection.Coordinates / RGBChoose Z-up or Y-up orientation and whether points carry sampled image colors.

      Do not treat point-cloud dimensions as real measurements unless you have external camera calibration and a metric depth source.

      The engine status orb summarizes the local FastAPI backend. Ready means inference endpoints respond. Degraded means the backend is reachable but diagnostics found missing or impaired capabilities. Offline means the renderer cannot reach the configured backend URL.

      • The backend URL defaults to 127.0.0.1:8765, or an Electron-provided local port when packaged.
      • Settings can change theme, motion, compact/high-contrast UI, backend URL, diagnostic refresh, cache badges, large-file warnings, webcam privacy behavior, logs, and navigation persistence.
      • Clearing frontend metrics affects session UI counters; clearing backend cache affects stored inference results.
      • Images and webcam frames flow through the local renderer/backend. DepthLens Pro does not require cloud inference; privacy changes only when you export, copy, or share files yourself.
